# /process-diag

Строит визуальную HTML-карту бизнес-процессов и архитектуры проекта: flowchart
по каждому домену, ER-диаграмма ключевых моделей, карта связей между доменами.
Использует шаблон и инструкцию из репозитория process_diag.

Этот файл — копия skill'а, который должен быть установлен в
`~/.claude/skills/process-diag/SKILL.md`, чтобы Claude Code мог вызывать его
как `/process-diag`. См. `README.md` в корне репозитория — раздел
"Установка skill" — как это сделать после клонирования.

## Usage

```
/process-diag <путь-к-проекту>
```

Если путь не указан — использовать текущую директорию.

## Что делать при вызове

### Шаг 0 — граф должен существовать

Проверить `<путь>/graphify-out/graph.json`. Если его нет — запустить
`/graphify <путь>` (полный пайплайн из skill graphify), дождаться завершения,
и только потом продолжать. Не пытаться строить диаграммы без графа —
без него нет материала для группировки по доменам.

### Шаг 1 — запустить build.sh

```bash
<repo>/build.sh <путь-к-проекту>
```

где `<repo>` — путь к клонированному репозиторию `process_diag` на этой
машине (см. установку в README). Это проверит граф ещё раз и соберёт
`<repo>/template.html` (с инлайн `mermaid.min.js`) в
`<путь-к-проекту>/<project>-processes.html`.

### Шаг 2 — выполнить инструкцию заполнения

Прочитать `<repo>/INSTRUCTIONS.md` и выполнить её целиком:
1. Прочитать `GRAPH_REPORT.md` и `graph.json` в `<путь>/graphify-out/`.
2. Определить домены. Для каждого домена — **обязательно** прочитать
   напрямую 3–6 реальных файлов (models + services), независимо от того,
   насколько подробным кажется граф — граф даёт структуру, не бизнес-смысл.
   Главный риск здесь — дописать типовой ERP/CRUD-паттерн (расчёт
   потребности → резервирование → утверждение → проверка на норму) вместо
   того, что реально в коде: если для шага flowchart нельзя назвать
   конкретный файл/класс/метод — это домысел, а не шаг из этого проекта
   (см. предупреждение в INSTRUCTIONS.md, Шаг 1).
3. Написать mermaid-код для каждого домена + карту доменов + ER-диаграмму,
   **строго соблюдая правило переносов строк** из INSTRUCTIONS.md (настоящий
   перевод строки внутри кавычек лейбла, никогда `\n` буквально и никогда
   `<br/>`).
4. Заполнить скопированный `<project>-processes.html`, заменив все
   `{{PLACEHOLDER}}` реальным содержимым.
5. Перед сдачей — самопроверка на домыслы из INSTRUCTIONS.md Шаг 5: для
   каждого узла flowchart и каждой модели убедиться, что можешь назвать файл
   и класс/метод/поле, где это подтверждается кодом.

### Шаг 3 — проверка

Если доступен инструмент `Artifact` — опубликовать готовый файл и убедиться
визуально (можно спросить пользователя, если сам не можешь проверить рендер),
что диаграммы отрисовались, а не остались как сырой текст с ошибками. Если
`Artifact` недоступен — просто сообщить пользователю путь к готовому файлу,
который можно открыть в браузере.

### Шаг 4 — отчёт пользователю

Кратко: сколько доменов, где лежит итоговый файл, что показывает карта
доменов и ER-диаграмма. Не пересказывать все диаграммы построчно.
