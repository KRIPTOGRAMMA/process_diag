# Инструкция: заполнение template.html диаграммами процессов

Это инструкция для агента (LLM), а не автоматический скрипт. `build.sh` только
проверяет предпосылки и копирует шаблон — вся содержательная работа (понять
процессы, сгруппировать по доменам, написать mermaid-код) делается здесь, руками
агента, читающего граф кодовой базы.

## Когда это применимо

Пользователь просит визуальную схему процессов/архитектуры проекта, и в
`<project>/graphify-out/graph.json` уже есть построенный graphify-графом (если
нет — сначала попросить пользователя или самому выполнить `/graphify <project>`).

## Шаг 1 — понять домены

1. Прочитать `<project>/graphify-out/GRAPH_REPORT.md`, если он существует.
   Там уже есть: список сообществ (communities) с размерами и метками, God
   Nodes, Surprising Connections, Suggested Questions — это готовая карта
   для ориентирования, не нужно заново кластеризовать вручную.
2. Прочитать `<project>/graphify-out/graph.json` — структура: `nodes` (id,
   label, file_type, source_file, community) и `edges` (source, target,
   relation, confidence).
3. Определить 5–10 крупнейших доменов (по числу узлов сообщества или по
   смысловой группировке файлов/директорий верхнего уровня). Для каждого
   домена найти его сервисный/бизнес-логический слой — файлы, где
   происходят вызовы (`services/`, основные модели) — не тесты и не
   сериализаторы, если явно не попросили.
4. Если графа недостаточно детально (граф даёт структуру вызовов, но не
   бизнес-смысл шагов) — прочитать напрямую 3–6 ключевых файлов на домен
   (сервисы + модели), чтобы восстановить реальную последовательность шагов,
   точки проверок и внешние системы. Не додумывать шаги, которых нет в коде.

## Шаг 2 — для каждого домена собрать три вещи

1. **Короткое summary** (1 предложение) — что это за процесс и почему он важен.
2. **Flowchart** — реальная последовательность шагов с точками ветвления
   (проверки/гейты) и внешними системами. Не выдумывать шаги — только то,
   что видно в коде (вызовы сервисов, проверки, if/else с бизнес-смыслом).
3. **Ключевые модели** — 2–4 модели с их важнейшими полями/FK, которые
   стоит знать, чтобы понимать процесс.

## Шаг 3 — mermaid-код

### Flowchart (для каждого домена)

```
flowchart TD
  A["Шаг 1<br/>детали"] --> B{"Точка проверки?"}
  B -- нет --> B1["Итог при отказе"]
  B -- да --> C["Следующий шаг"]

  classDef gate fill:#fbeee0,stroke:#a85a1e,color:#1a1d24;
  classDef stop fill:#f4d9d9,stroke:#a83a3a,color:#1a1d24;
  class B gate
  class B1 stop
```

**КРИТИЧЕСКИ ВАЖНО про переносы строк внутри узлов:**
- НЕ используй `\n` (буквальный backslash-n) — mermaid внутри `<pre>` не
  интерпретирует escape-последовательности, текст напечатается как есть
  (`\n` появится как два символа на экране).
- НЕ используй `<br/>` внутри `["..."]`/`{"..."}` строковых лейблов — в
  этой версии mermaid тег не рендерится и слова слипаются без пробела.
  Это касается **обоих** типов узлов — и прямоугольных `["..."]`, и
  гейтов-ромбов `{"..."}` (баг воспроизводился именно на гейтах, где часто
  забывают об этом правиле, т.к. вопросительный текст короткий и кажется
  безопасным сократить его через `<br/>`):
  ```
  ❌ D{"Заказ<br/>корректен?"} --> ...   слова слипнутся: "Заказкорректен?"
  ✅ D{"Заказ
  корректен?"} --> ...                  настоящий перенос строки — работает
  ```
- Используй **настоящий перевод строки** прямо внутри кавычек лейбла:
  ```
  A["Первая строка
  Вторая строка"] --> B
  ```
  Mermaid поддерживает многострочные quoted-лейблы нативно — это
  единственный проверенный рабочий вариант.

Проверенная цветовая схема гейтов/стопов (совпадает с CSS темы шаблона,
работает в обеих темах за счёт мягких полупрозрачных тонов):
- `classDef gate fill:#fbeee0,stroke:#a85a1e,color:#1a1d24;` — точка решения
- `classDef stop fill:#f4d9d9,stroke:#a83a3a,color:#1a1d24;` — отказ/стоп
- `classDef ext fill:#eaf0e6,stroke:#5b6b52,color:#1a1d24;` — внешняя система
- `classDef hub fill:#e8edf7,stroke:#3a5ba0,color:#1a1d24;` — центральный/hub узел

### Карта доменов (одна на всю страницу)

```
flowchart TB
  DomainA["01 · Домен A"]
  DomainB["02 · Домен B"]
  DomainA -- "создаёт X" --> DomainB
  classDef hub fill:#e8edf7,stroke:#3a5ba0,color:#1a1d24;
  class DomainB hub
```

Показывает верхнеуровневые связи между доменами (не шаги внутри процесса) —
что от чего зависит, куда идут деньги/данные, какой домен центральный.

### ER-диаграмма (одна на всю страницу)

```
erDiagram
  ORGANIZATION ||--o{ FILIAL : "имеет"
  FILIAL ||--o{ MODEL_X : "имеет"
  MODEL_X ||--|| MODEL_Y : "создаёт"
```

Используй реальные кардинальности (`||--o{`, `}o--||`, `||--||`) исходя из
FK/relations в моделях, не приблизительно.

## Шаг 4 — заполнение template.html

Плейсхолдеры в `template.html`:

| Плейсхолдер | Что подставить |
|---|---|
| `{{PAGE_TITLE}}` | Заголовок вкладки, например `Карта процессов <project>` |
| `{{PROJECT_LABEL}}` | Эйcброу над названием в сайдбаре, напр. `<project> · внутренняя документация` |
| `{{PAGE_H1_SHORT}}` | Короткое название в сайдбаре, напр. `Карта процессов` |
| `{{EYEBROW}}` | Строка статистики, напр. `apps/ · N доменов · M узлов графа кода` |
| `{{PAGE_H1}}` | Заголовок h1 на странице |
| `{{PAGE_INTRO}}` | 1–2 предложения о том, что показывает страница |
| `{{NAV_LINKS}}` | `<a href="#domain-id"><span class="n">01</span>Название</a>` на каждый домен + `<a href="#map">Карта доменов</a>`, `<a href="#er">ER · ключевые модели</a>` |
| `{{DOMAIN_SECTIONS}}` | Полный HTML всех доменных секций (см. образец ниже) |
| `{{MAP_SECTION}}` | Секция "Карта доменов" с flowchart |
| `{{ER_SECTION}}` | Секция ER с erDiagram |
| `{{PATTERNS_SECTION}}` | (опционально) секция сквозных паттернов, если они есть — иначе можно оставить пустым или удалить весь блок |

### Образец одной доменной секции

```html
<section class="domain" id="visits">
  <div class="domain-head"><h2><span class="idx">01</span>Название домена</h2></div>
  <p class="domain-summary">Одно предложение о процессе.</p>

  <div class="diagram">
    <pre class="mermaid">
flowchart TD
  ...
    </pre>
    <div class="diagram-caption">flowchart · короткое описание</div>
  </div>

  <h3>Ход процесса</h3>
  <div class="flow">
    <div class="step">
      <div class="step-title"><span class="actor">роль</span>Название шага</div>
      <div class="step-body">Описание с <code>именами функций</code> и <span class="field">полями</span>.</div>
    </div>
    <div class="step gate">
      <div class="step-title"><span class="actor">проверка</span>Точка решения</div>
      <div class="step-body">...</div>
    </div>
  </div>

  <h3>Ключевые модели</h3>
  <div class="panel-grid">
    <div class="panel model">
      <h4>ModelName</h4>
      <dl>
        <dt>field</dt><dd class="fk">→ OtherModel</dd>
      </dl>
    </div>
  </div>
</section>
```

Используй классы `.step` (обычный шаг), `.step.gate` (точка проверки —
жёлтая точка слева), `.step.external` (внешняя система — зелёная точка).

## Шаг 5 — проверка перед сдачей

- **Перед публикацией** прогнать по файлу:
  `grep -c '<br/>' <project>-processes.html` — если результат больше 0,
  найти эти вхождения (`grep -n '<br/>' ...`) и убедиться, что они лежат
  **только** внутри инлайн-скрипта mermaid.min.js (минифицированный JS,
  далеко от `<pre class="mermaid">` блоков), а не в самих диаграммах,
  которые заполнял агент. Любой `<br/>` внутри `<pre class="mermaid">...
  </pre>` — баг, надо заменить на настоящий перенос строки (см. Шаг 3).
- Открыть готовый `<project>-processes.html` (через `Artifact` инструмент
  Claude Code, если пользователь работает в этой среде, либо просто в
  браузере) и убедиться, что диаграммы реально отрендерились, а не остались
  как текст/ошибка. Отдельно проверить хотя бы один гейт (ромб `{"..."}`) —
  именно там чаще всего проскакивает `<br/>` по ошибке.
- Проверить обе темы (светлая/тёмная), если пользователь может переключать.
- Проверить, что нет буквальных `\n` в узлах диаграмм (см. Шаг 3).
- Не оставлять неразрешённые `{{PLACEHOLDER}}` в финальном файле.
- Открывать файл напрямую (`file://`) или через `Artifact` — не через
  dev-server с live-reload (см. README, "Как открывать готовый файл").
